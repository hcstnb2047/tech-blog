## この記事でやること

スマートフォンから自宅（または常時起動の）サーバーにSSHで入り、その上で動くAIコーディングエージェント Claude Code を操作して、開発作業を進める。この構成の作り方を最短で示す。

向いているのは、机にまとまって向かう時間を取りづらく、移動中やちょっとした空き時間にも作業を一歩進めたい人だ。重い処理はサーバー側に置き、手元のスマホは「入力と確認の窓」に徹する。画面は小さいが、エージェントに指示を出して結果を確認する用途なら十分に成立する。

## 構成：4つの部品とそれぞれの役割

この構成は4つの部品と1つのエージェントでできている。役割を先に押さえると、後の手順が「なぜそれをやるのか」で読める。

| 部品 | 役割 |
|------|------|
| **Tailscale** | スマホとサーバーを同じ仮想ネットワークに入れる。外出先からでも、固定IPやポート開放なしに繋がる |
| **SSH** | サーバーへ安全にログインし、ターミナルを手元に持ってくる |
| **tmux** | サーバー側のセッションを生かし続ける。回線が切れても作業が消えない |
| **ターミナルアプリ** | スマホ側のSSHクライアント（iOSなら Termius など、Android なら Termux など） |
| **Claude Code** | サーバー上で動く本体。指示を受けてファイルを編集・コマンド実行・テストまで行う |

このうち**いちばん重要なのは tmux**だ。理由は次に述べる。

## キモは tmux：回線が切れても作業が死なない

スマホのSSHは、画面を切り替えたり電波が弱くなったりすると、すぐ接続が切れる。普通にコマンドを実行している最中に切れると、その処理ごと止まってしまう。

tmux はサーバー側で動く「セッションの保存箱」だ。作業を tmux の中で動かしておけば、スマホの接続が切れても**処理はサーバー側で走り続ける**。電波が戻ったら、同じセッションに繋ぎ直すだけで、切れる前の画面がそのまま出てくる。

```bash
# サーバー側で、なければ作る・あれば繋ぎ直す（これ1つでよい）
tmux new-session -A -s main
```

`-A` は「同名のセッションがあれば繋ぎ直し、なければ新規作成」を1コマンドにまとめるオプション。スマホから入り直すたびにこれを打てば、常に同じ作業場に戻れる。Claude Code はこの tmux の中で起動しておく。

## ネットワークは Tailscale で繋ぐ

外出先のスマホから自宅サーバーに繋ぐには、本来ならグローバルIPやポート開放、VPNの設定が要る。Tailscale はこの面倒をほぼ消す。

両方の端末に Tailscale を入れて同じアカウントでログインすると、2台が同じ仮想ネットワーク（プライベートな閉じた網）に入る。あとはサーバーに割り当てられた Tailscale 上のアドレスへSSHするだけ。ポート開放をしないので、インターネットに直接さらされる入り口を作らずに済む。

ここで混同しやすい点を一つ。**Tailscale は SSH ではなく、WireGuard を土台にしたVPN（ネットワークの層）**だ。この記事の構成は「Tailscale で繋いだネットワークの上で、通常の SSH を流す」形で、両者は別の層になる。Tailscale が繋ぐところまでをやり、シェルを取りに行くのは SSH の仕事、という分担だ。

なお紛らわしいことに、Tailscale には「Tailscale SSH」という別機能もある。これは Tailscale 側が認証・暗号化を肩代わりし、SSH鍵の手動管理を不要にするものだ。鍵管理を省きたい場合の選択肢になるが、本記事では通常の SSH を前提にする。

## 最短の手順

1. **サーバーとスマホの両方に Tailscale を入れ**、同じアカウントでログインする
2. **サーバーで SSH を有効化**しておく（多くのLinux/macOSは標準で持っている）
3. **スマホにターミナルアプリ**（Termius など）を入れ、接続先に Tailscale 上のサーバーアドレスを登録する
4. スマホからSSHでサーバーに入り、`tmux new-session -A -s main` でセッションに入る
5. その中で **Claude Code を起動**し、普段通り指示を出す
6. 作業を中断するときは、tmux を切るだけ（`Ctrl-b` の後に `d` でデタッチ）。スマホを閉じても処理は続く
7. 戻ってきたら、またSSHして同じコマンドで繋ぎ直す

これだけで「どこからでも同じ作業場に戻れる」状態になる。

## 実際の使い勝手：向くこと・向かないこと

淡々と現実を書いておく。

**向くこと**

- エージェントへの指示出しと、出てきた結果の確認
- 短いバグ修正、調査、コードレビュー
- 走らせておきたい長い処理を投げて、後で結果を見る使い方（tmux があるので放置できる）

**向かないこと・つらいこと**

- 大量のコードを手で打ち込む作業（スマホのキーボードは不利）
- 細かいGUI操作が要るもの（そもそもターミナルだけの世界）
- 画面いっぱいに情報を広げて見比べる作業

つまり「手を動かして書く」より「任せて読む・確かめる」用途に寄せると、この構成は素直に効く。エージェントに委任する作業との相性がいい、と言い換えてもいい。

## 「リモート実行があるのに、自前サーバーは要る?」

公式にエージェントをリモート/クラウドで動かす仕組みも整ってきている。「それを使えば、わざわざ自分でサーバーを立てる必要はないのでは?」という意見は当然ある。エージェントに作業をさせるだけなら、その指摘はおおむね正しい。

それでも自前サーバーを置く構成が効くのは、**エージェントの担当範囲の外まで管理したい場合**だ。たとえば次のような用途は、用意されたリモート実行の枠だけでは収まりにくい。

- 定期実行のバッチ（cron）を自分で組んで回す
- ログやデータファイルを直接置いて管理する
- エージェントの守備範囲を超えた、サーバーそのものの設定・運用

サーバーを1台持つということは、エージェントの実行環境だけでなく**マシン全体を手元に持つ**ということだ。エージェント以外の常時稼働の仕組みも同じ場所に同居させられる。ここに価値を感じるなら、リモート実行があっても自前サーバーは無駄にならない。

正直な補足を一つ。この構成を組んだ時点では、そもそも公式のリモート実行という選択肢が無かった、という事情もある。先に自前で作った結果、いまもバッチ運用などで使い続けている、というのが実際のところだ。「今ゼロから始めるなら何を選ぶか」と「すでにある仕組みを活かすか」は別の問いとして分けて考えるとよい。

## まとめ

- **Tailscale**で繋ぎ、**SSH**で入り、**tmux**で生かし続け、その中で**Claude Code**を動かす
- 一番のキモは tmux。回線が切れても作業が死なないから、スマホの不安定さを気にせず使える
- 手で大量に書く用途には向かないが、エージェントへの指示と確認なら、スマホだけでも作業は十分に前へ進む

机に向かう時間がまとまって取れなくても、空いた時間に一歩ずつ進められる作業環境は、この最小構成で作れる。
