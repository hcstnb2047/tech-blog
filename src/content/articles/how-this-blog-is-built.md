このブログは「**読むのが速い静的サイト**」と「**うっかり公開を機械で止める仕組み**」の二本柱で組んでいる。凝った機能はなく、構成は意図的に薄い。技術記事を出し続けるうえで最優先したのは、華やかさではなく「事故らずに続けられること」だからだ。

この記事では、その構成と、なぜその技術を選んだのかをまとめる。

## 全体像

ざっくり3つの部品でできている。

- **公開サイト**: Markdown の記事を静的生成して配信する面
- **安全ゲート**: 公開リポジトリへの変更を毎回スキャンし、個人情報や秘密情報が混ざっていたら公開を止める仕組み
- **下書き環境**: 記事の素材と下書きは、公開リポジトリとは物理的に分けた別の非公開リポジトリで管理する

「書く場所」と「出す場所」を分け、その境目に機械的な検問所を置く。これが設計の中心にある考え方だ。

## 公開サイト: Astro

公開面は Astro で作っている。Astro はコンテンツ志向の静的サイトジェネレータで、**既定でブラウザに JavaScript をほとんど送らない**。記事を読むだけのサイトに重いランタイムは要らないので、表示は速く、検索エンジンにも乗りやすい。

記事の入れ物には Astro の Content Collections を使っている。ここが地味に効いていて、**記事のメタ情報（タイトル・説明・公開日・出典）をビルド時に型で検証**できる。スキーマはこれだけだ。

```ts
schema: z.strictObject({
  title: z.string(),
  description: z.string(),
  publishedAt: z.string().date(),
  tags: z.array(z.string()).default([]), // 一覧での分類用（任意）
  sources: z.array(z.object({ title: z.string(), url: z.string().url() })).min(1),
})
```

ポイントは `strictObject` と `sources(...).min(1)` の2点だ。`strictObject` はスキーマに無いキーを拒否するので、**個人的な記録由来の余計なフィールドが紛れ込むとビルドが失敗する**。`min(1)` は出典ゼロの記事を弾く。「裏取りのない記事を公開しない」という方針を、運用ルールでなく仕組みで縛る狙いだ。

`tags` のような表示用フィールドを後から足すときも、このスキーマに明示的に宣言して初めて通る。**「許可したものだけ載る」状態（allowlist）を保ったまま機能を増やせる**ので、フィールド追加は事故ではなく毎回の意思決定になる。

## 見た目の土台: Tailwind v4 と Expressive Code

スタイリングは Tailwind CSS v4 を `@tailwindcss/vite` 経由で入れている（2026 時点の推奨構成）。色や余白などのデザイントークンは CSS の `@theme` に集約し、ライト/ダークの切り替えは `prefers-color-scheme` に任せている。配色が変わってもスクリプトは動かないので、**テーマ切り替えでクライアント JavaScript は増えない**。

記事本文は `@tailwindcss/typography`（prose）でまとめて整形し、コードブロックは Expressive Code に任せている。テーマ・行ハイライト・コピーボタン・ファイル名表示まで、技術記事で要る体裁が最初から揃う。Expressive Code のコピーボタンだけは小さな JS が付くが、これはフレームワークのランタイムではなく、**本文を読むのに JS を要求しない**という方針は崩していない。

見出しの「#」リンクのような細かな機能も、新しい依存を足さずに付けている。Astro が見出しに採番する id をそのまま使い、生成済みのツリーを歩いて静的なリンクを差し込むだけの小さな処理にした。**「機能は足すが、依存とランタイム JS は増やさない」**を細部まで通すのが、軽さを保つための作法だ。

そして、こうしたデザイン上の判断は `DESIGN.md` という1枚に集約している。トークンの値・読み幅・余白スケール・なぜその選択にしたかを言葉で残し、**見た目のルールが各ファイルに散らばって理由が消えるのを防ぐ**。スタイルの単一ソースを1か所に置くこと自体を、開発手法の一部にしている。

## 安全ゲート: CI でのスキャン

スキーマが守れるのは記事の「メタ情報」までで、**本文に書いてしまった個人情報までは止められない**。そこを埋めるのが2段目のゲートだ。

公開リポジトリへ変更を反映するたびに、CI で秘密情報スキャナ（Gitleaks）を走らせている。API キーやトークンのような汎用シークレットに加えて、**このブログ固有のカスタム検出ルール**を足してある。検出すればその時点で**ビルドを失敗させ、公開を止める**。警告で素通りさせず、必ず落とす設定にしているのが要だ。

ここは作っただけでは不十分だ。わざと検出対象を仕込んだダミー記事を流し、**本当にブロックされること**を確認してから本運用に入れている。ゲートは「効くと信じる」ものではなく「効くと確かめる」ものだという考え方だ。

ただしこの機械ゲートにも限界がある。既知のパターンは捕まえられるが、言い換えや文脈で滲み出る情報までは正規表現では拾いきれない。だから最終的な「これは出していいか」の判断は人間が持つ、という前提を崩さない設計にしている。

## 非公開と公開を物理的に分ける

下書きや素材は、公開リポジトリとは別の非公開リポジトリに置いている。健康関連を含む個人的な記録もそこにあるので、**公開リポジトリには最初からそれらを持ち込まない**設計にした。公開側のリポジトリは履歴を引き継がず、クリーンな状態から始めている。最新版で消しても履歴から復元できる、という事故を避けるためだ。

公開は自動化していない。下書きから公開面へ移すときは、明示的にコピーし、差分を人の目で確認してから出す。手数は増えるが、不可逆な操作の手前には人間の確認を1つ残す方針だ。

## ホスティング: Vercel

配信は Vercel に載せている。Astro の静的出力をそのままデプロイでき、初期構築の手間が小さい。独自ドメインや凝った計測は後回しにして、まずは「URL を踏めば読める」状態を最短で作ることを優先した。

## まとめ

この構成は、技術的に新しいことは何もしていない。やったのは、**よくある部品を「安全に公開する」という一点に向けて並べ直した**ことだけだ。書く場所と出す場所を分け、境目に機械の検問と人間の確認を置く。続けるための土台としては、これで十分だといえる。
