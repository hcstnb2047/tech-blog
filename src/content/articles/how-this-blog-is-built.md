このブログは「**読むのが速い静的サイト**」と「**うっかり公開を機械で止める仕組み**」の二本柱で組んでいる。凝った機能はなく、構成は意図的に薄い。技術記事を出し続けるうえで、私が一番欲しかったのは華やかさではなく「事故らずに続けられること」だったからだ。

この記事では、その構成と、なぜその技術を選んだのかをまとめる。

## 全体像

ざっくり3つの部品でできている。

- **公開サイト**: Markdown の記事を静的生成して配信する面
- **安全ゲート**: 公開リポジトリへの変更を毎回スキャンし、個人情報や秘密情報が混ざっていたら公開を止める仕組み
- **下書き環境**: 記事の素材と下書きは、公開リポジトリとは物理的に分けた別の非公開リポジトリで管理する

「書く場所」と「出す場所」を分け、その境目に機械的な検問所を置く。これが設計の中心にある考え方だ。

## 公開サイト: Astro

公開面は Astro で作っている。Astro はコンテンツ志向の静的サイトジェネレータで、**既定でブラウザに JavaScript をほとんど送らない**。記事を読むだけのサイトに重いランタイムは要らないので、表示は速く、検索エンジンにも乗りやすい。

記事の入れ物には Astro の Content Collections を使っている。ここが地味に効いていて、**記事のメタ情報（タイトル・説明・公開日・出典）をビルド時に型で検証**できる。スキーマはこれだけだ。

```ts
schema: z.strictObject({
  title: z.string(),
  description: z.string(),
  publishedAt: z.string().date(),
  sources: z.array(z.object({ title: z.string(), url: z.string().url() })).min(1),
})
```

ポイントは `strictObject` と `sources(...).min(1)` の2点だ。`strictObject` はスキーマに無いキーを拒否するので、**個人的な記録由来の余計なフィールドが紛れ込むとビルドが失敗する**。`min(1)` は出典ゼロの記事を弾く。「裏取りのない記事を公開しない」という方針を、運用ルールでなく仕組みで縛りたかった。

## 安全ゲート: CI でのスキャン

スキーマが守れるのは記事の「メタ情報」までで、**本文に書いてしまった個人情報までは止められない**。そこを埋めるのが2段目のゲートだ。

公開リポジトリへ変更を反映するたびに、CI で秘密情報スキャナ（Gitleaks）を走らせている。API キーやトークンのような汎用シークレットに加えて、**自分用のカスタム検出ルール**を足してある。検出すればその時点で**ビルドを失敗させ、公開を止める**。警告で素通りさせず、必ず落とす設定にしているのが要だ。

正直に書くと、ここは作っただけでは安心できなかったので、わざと検出対象を仕込んだダミー記事を流し、**本当にブロックされること**を確認してから本運用に入れた。ゲートは「効くと信じる」ものではなく「効くと確かめる」ものだと思っている。

ただしこの機械ゲートにも限界がある。既知のパターンは捕まえられるが、言い換えや文脈で滲み出る情報までは正規表現では拾いきれない。だから最終的な「これは出していいか」の判断は人間が持つ、という前提を崩さないようにしている。

## 非公開と公開を物理的に分ける

下書きや素材は、公開リポジトリとは別の非公開リポジトリに置いている。健康関連を含む個人的な記録もそこにあるので、**公開リポジトリには最初からそれらを持ち込まない**設計にした。公開側のリポジトリは履歴を引き継がず、クリーンな状態から始めている。最新版で消しても履歴から復元できる、という事故を避けるためだ。

公開は自動化していない。下書きから公開面へ移すときは、明示的にコピーし、差分を自分の目で見てから出す。手数は増えるが、不可逆な操作の手前には人間の確認を1つ残しておきたい。

## ホスティング: Vercel

配信は Vercel に載せている。Astro の静的出力をそのままデプロイでき、初期構築の手間が小さい。独自ドメインや凝った計測は後回しにして、まずは「URL を踏めば読める」状態を最短で作ることを優先した。

## まとめ

この構成は、技術的に新しいことは何もしていない。やったのは、**よくある部品を「安全に公開する」という一点に向けて並べ直した**ことだけだ。書く場所と出す場所を分け、境目に機械の検問と人間の確認を置く。続けるための土台としては、これで十分だと今は考えている。
